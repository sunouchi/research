# 記事作成フロー — トレース記録

このドキュメントは、デザイン野帳の記事作成タスクで実際に使ったフローを後から辿れるように記録する場所。**「次回もこの通り」とは限らない** ── 各回の状況に応じて見直す前提だが、何をどう進めたかを参照できるようにする。

新しい記事作成サイクルを実施した場合は、末尾に「## ログ」セクションを追加する形で追記する。

---

## 標準フロー（vol 4 / vol 5 のサイクルから抽出）

### Phase 0：前提整理

新シリーズの執筆を始める前に、以下を確認・合意する：

- 対象の本／著者／テーマ
- 何回構成にするか（複数記事に分けるかどうか、暫定）
- 想定読者（具体的に）
- 各記事の想定字数（CLAUDE.md 既定：3000〜4000 字）
- 文章形式の選択（エッセイ／レポート／その他）
- 文章構成（起承転結／自由構成／7 セクション等）
- 形式上の制約（暗喩的に触れる話題、避ける話題、etc.）
- 各記事の独立性／二部作扱いの可否（CLAUDE.md 既定：基本独立）

→ 不明な点は AskUserQuestion で 3〜7 問程度に整理して確認する。

### Phase 1：横断リサーチ（並列サブエージェント）

シリーズ全体で再利用できる **横断資料**（人物伝記、本の構造マップ、概念の語源、関連系譜、特定概念の深掘り、etc.）を、サブエージェントで**並列に**作成する。

- 配置：`yacho/notes/<シリーズslug>/`
- 1 エージェント＝ 1 トピックの単一目的タスクとして切る（汎用すぎる依頼は出力品質が落ちる）
- 各エージェントには CLAUDE.md 方針（個人情報禁止、PDF 扱い、長文転載禁止）と出力先パスを明示
- 公開ウェブソース中心（出版社ページ、SEP/IEP、Wikipedia、研究者個人サイト、雑誌記事）
- 並列数：実用上 4〜6 並列が扱いやすい

### Phase 2：執筆方針の対話（AskUserQuestion）

リサーチ素材をもとに、各記事の中心テーマと形式上の細部を**ユーザーと対話して**決める。AskUserQuestion を 4 問単位で使う（仕様上 1 回最大 4 問）。

確認すべき主な軸：
- 各記事の中心テーマ／概念（1 つに絞るのが基本）
- 記事間の関係（独立／緩い連続／二部作）
- 暗喩的扱いの話題の重み付け（例：AI 史を体系的に置くか暗喩的に留めるか）
- 副読資料としての立ち位置（補助線型／ガイドブック型／誘導型）
- デザイン実践への含意の書き方（明示／allusive／書かない）
- 起の入り口（時代背景／概念パズル／個人エピソード／読者への問い）

### Phase 3：アウトライン作成

確定した方針をもとに、各記事のアウトラインを書く。

- 配置：`yacho/notes/N-YYYYMMDD-outline.md`（過去シリーズと同じ慣習）
- 内容：暫定タイトル候補、想定分量、想定読者、形式の方針、各段（起承転結等）の核と書き手のメモ、構成メモ、参照ルール、引き継ぐ問い（公開記事には書かない作業メモ）
- 複数記事を並列で執筆する場合は、アウトラインも並列で作成可能

### Phase 4：形式の追加検討（必要に応じて）

エッセイ形式に加えて他の形式（レポート、対話、書評など）も並走させる選択肢が出る場合がある。その場合は別途 AskUserQuestion で形式の細部を詰める：

- 章立て構造
- 想定字数
- 参照文献の表記スタイル（本文中軽く言及＋末尾一覧／番号脚注／ハーバード式 等）
- 内容範囲（エッセイと同じ／広め／狭め深め）

### Phase 5：ドラフト生成（並列サブエージェント推奨）

各記事のドラフトを生成する。

- 配置：`yacho/notes/<シリーズslug>/article-draft/`
  （`yacho/manuscript/` は note.com に**公開する原稿**のみ。候補・ドラフトはここに置かない）
- ファイル名規約：`N-YYYYMMDD-<format>.md` 形式（例：`4-20260524-essay.md`, `4-20260524-report.md`）
- バージョン分けしたファイル（`..._v2.md`, `..._final.md`）は作らない。同一ファイルを直接編集（CLAUDE.md 方針）
- **複数ドラフトの同時生成は並列サブエージェント化する**（CLAUDE.md 方針）
- 各エージェントには、対応するアウトライン、横断リサーチノートへの参照、CLAUDE.md 方針、出力先パスを明示

### Phase 6：形式の選択 → ブラッシュアップ

ユーザーが採用形式を選んだら、選ばれた形式に絞ってブラッシュアップ。

- 不要セクションの削除
- 主要概念や参照文献へのリンク付与（公開を意識した整え）
- リンク作業はサブエージェント化し、**「リンクを設定するタスク」と「リンクが正しいかチェックするタスク」を分離**して実行する（互いに依存しないチェック視点を確保）
- **Wikipedia は英語版（en.wikipedia.org）を優先する**。日本語版（ja.wikipedia.org）の情報量は英語版に比べて薄いことが多いため、英語版を既定とする。例外として、日本語版が同等以上に充実している場合のみ ja 版を使う（典型例：日本の学術伝統が深い領域 — 仏教概念、日本史、和歌・俳句など）
- リンクの優先順：Wikipedia（上記ルール）→ SEP/IEP → 著者公式 → 出版社公式
- リンクを捏造しない（実在検証を必ず行う）
- 見つからない場合はリンクなしで残す

### Phase 7：公開準備（必要に応じて）

- ドラフトを `yacho/manuscript/` にコピー／移動して最終整え
- note.com 投稿
- `yacho/roadmap.md` の該当回欄を更新（公開予定日と URL）

### Git 運用（CLAUDE.md より）

- ブランチ命名：`yacho/vol-NN-<short-slug>`
- 記事執筆の節目で push → PR → main にマージ
- コミット粒度：運用ルール変更／原稿そのもの／参考素材 などレビューしやすい単位

## サブエージェント運用の指針

### 並列化の判断基準

- **独立タスクで、出力先が異なる** → 並列化する
- **依存関係がある**（A の出力が B の入力）→ 順次実行
- **同じファイルを編集する**複数タスク → 順次実行（編集競合回避）

### サブエージェントへの依頼の書き方

各エージェントが自己完結で作業できるよう、以下を明示する：

1. 役割と背景（何のためのタスクか、文脈）
2. 対象ファイルパスと現状
3. 具体的な作業内容（before / after 例を入れると精度が上がる）
4. 厳守事項（捏造禁止、本文改変禁止、出力先、CLAUDE.md 方針への参照）
5. 効率化のヒント（並列 WebFetch、二段構え検索 等）
6. 報告のフォーマット（追加件数、未対応項目とその理由）

### 観察された落とし穴

- **逐次実行になりやすい**：複数ドラフトを書く際、主体エージェント（私）が自分で書いてしまい並列化を逃す。CLAUDE.md ルール追加（並列処理優先）の理由
- **リンク捏造**：WebFetch せずに「ありそうな URL」を生成してしまう。捏造禁止を明示する必要がある
- **同名異物リンク**：「Structural coupling」がソフトウェア工学項目にリダイレクトする等。検証ステップで弾く
- **ja.wikipedia 既定の弊害**：「日本語ページがあるからまず ja に貼る」をデフォルトにすると、ja 版が極端に薄い場合（例：研究者個人ページ、専門概念）に読者の navigate 体験が貧弱になる。en 版を既定にし、ja 版が同等以上の場合のみ ja を使うルールに変更（vol 5 公開準備時に発覚）
