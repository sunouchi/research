# このリポジトリについて

個人の研究活動をオープンに共有するためのpublicリポジトリ。
「何を読み、何を考えているか」を公開することが主目的で、論文PDFそのものの配布が目的ではない（ただし `yacho/` 配下の重要論文は例外的にコミット対象に含める。後述）。

## 重要な方針

### 個人情報・機密情報をリポジトリに含めない
publicリポジトリである以上、以下は **一切書かない／コミットしない**：
- ローカルマシンの絶対パス（ユーザー名を含むパスなど）
- メールアドレス、氏名、電話番号等の個人情報
- APIキー、トークン、パスワード等のクレデンシャル
- 所属組織の非公開情報

Claudeがドキュメントやコードを書くときは、パスは相対パス（例: `dario/dario_papers/`）で表記すること。
ユーザー固有の環境に言及する必要がある場合は、`~/...` や `<repo-root>/...` などの抽象表記を用いる。

新しいファイルを作成・編集する際は、コミット前に上記が混入していないか確認する。

### PDFの扱い
- `*.pdf` / `*.PDF` は `.gitignore` で**原則除外**
- 理由: 出版社が著作権を持つ論文をpublicリポジトリに置くと規約違反の恐れがあるため
- **例外: `yacho/` 配下の PDF はコミット対象に含める**
  - デザイン野帳の活動で参照する重要論文を、リポジトリ内で完結して扱うための運用上の例外
  - `.gitignore` 側で `!yacho/**/*.pdf` / `!yacho/**/*.PDF` のネゲーションを入れている
  - フェアユース／引用範囲を超えた利用は避ける、という基本姿勢は変わらない
- それ以外（`dario/` など）の PDF は従来どおりローカルおよび Obsidian で管理し、リポジトリには置かない

### リポジトリに置いてよいもの
- 読書リスト・論文メモ（Markdown）
- 自分の要約・考察・引用（フェアユース範囲の短い引用はOK、長文転載はNG）
- `.bib` ファイル（書誌情報）
- 論文へのリンク（DOI、arXiv、出版社ページ）
- ダウンロード用スクリプト等（ただし著作物の再配布に該当しない形で）

## Claudeへの指示

### やってほしいこと
- 論文の内容について質問したら、ローカルのPDFを読んで要約・考察を返す
- 新しい論文メモを作るときは、既存ファイル（例: `dario/dario-papers-list.md`）のフォーマットに合わせる
- 著作権上グレーな操作（論文の全文転載、長い原文引用など）は事前に確認する

### やらないでほしいこと
- `yacho/` 配下**以外**の PDF ファイルを `git add` / コミット対象に含めること（`yacho/` 配下は例外的に可）
- 論文本文を長文でMarkdownに転記すること（短い引用＋自分の要約に留める）
- 自分の判断でリポジトリをpublic→privateに切り替えたり、その逆をしたりすること

## 作業環境
- PDFの同期先: ローカルPC ＋ Obsidian（iPhoneからも閲覧可）
- リポジトリ公開範囲: **public**

# デザイン野帳（yacho/）の運用ルール

`yacho/` 配下は、note に投稿しているシリーズ「デザイン野帳」の作業ディレクトリ。原稿の執筆・素材管理をここで行う。

## フォルダの役割

- **`yacho/draft/`** — 下書きの作業ファイル。アウトライン作成と粗い原稿はここで進める
- **`yacho/manuscript/`** — note に投稿する完成原稿。`draft/` の内容をもとに整える
- **`yacho/doc/`** — 複数の記事にまたがって参照する横断的な情報（書籍要約・概念整理など）
- **`yacho/paper/`** — デザイン野帳の活動で参照する**重要論文**の置き場。重要でない論文はここに置かない。**ここの PDF はコミット対象に含める**（リポジトリ全体方針の例外）
- **`yacho/roadmap.md`** — シリーズ全体の予定表

## 記事執筆時の参照ルール

第1フェーズではショーン・ギャラガーの身体性認知を扱う。**ギャラガー回の記事執筆タスクを受けた際は、回答前に必ず以下を読み込むこと**：

1. **ギャラガーの英語原典** — `yacho/paper/gall23cambridge.pdf`
   （*Embodied and Enactive Approaches to Cognition*, Cambridge University Press, 2023）
2. **その要約ノート** — `yacho/doc/gallagher-embodied-enactive-cognition-summary.md`

両方に目を通したうえで、原典の語彙・章立てに即した記述を心がける。要約ノートだけで答えない。

## 原稿執筆の進め方

- **記事の分量は 3000〜4000 字（日本語）を目安にまとめる**。`yacho/draft/` でアウトラインを作る段階から、この字数に収める前提で構成する
- フォーマット・トーンは既存の manuscript ファイル（例: `yacho/manuscript/1-20260426-embodied-cognition-v2.md`）に揃える
- **書かれている内容を全部漏れなく要約しようとしなくてよい。面白いと思うポイントを重点的に取り上げ、その面白さがどういうところにあるのかを、主観を適度に交えながら読者に伝える**。中立的な紹介文ではなく、読者の手を引いて「ここが面白い」と指し示す書き手の声を通す
- 各記事の末尾に「残った問い」を置き、次回への接続点を残す運用にしている
- 原稿に長文の原文転載は入れない（短い引用＋自分の要約に留める。リポジトリ全体方針と同じ）

## ファイル運用ルール

- **`yacho/manuscript/` および `yacho/draft/` 配下では、バージョン分けしたファイルを新規作成しない**
  - 例: `..._v2.md` `..._final.md` のような枝分かれは作らず、同一ファイルを直接編集する
  - 過去にそのような運用が残っていても、今後は踏襲しない
  - 履歴は git に任せる方針

## Git 運用（記事執筆フロー）

記事執筆は **main で作業せず、回ごとに専用ブランチを切って進める**。プルリクエストの単位で「1回の記事執筆で何が起きたか」を後から振り返れるようにするため。

- **ブランチ命名**: `yacho/vol-NN-<short-slug>` 形式（例: `yacho/vol-02-embodied-embedded-extended`）
  - `NN` は2桁ゼロ埋めの巻数、`<short-slug>` は記事テーマを表す英小文字ハイフン区切り
- **作業の流れ**:
  1. 記事執筆を始める時点で、main から該当回のブランチを切る
  2. 原稿（draft / manuscript）・必要に応じた参考論文（paper）・ロードマップ更新などを、そのブランチ上にコミットしていく
  3. コミットは適度な粒度で分ける（運用ルールの変更／原稿そのもの／参考素材 など、レビュー時に追いやすい単位）
  4. 記事公開の節目で push → プルリクエストを作成し、main にマージする
- **粒度の目安**: 1ブランチ＝1回分の記事。ロードマップだけの更新など、特定の回に紐づかない軽微な変更は main 直 commit でも可（リポジトリ全体の方針に従う）
- **コミットメッセージ**: 既存のスタイル（日本語、`yacho vol-NN ...` 接頭で何が変わったかを簡潔に）に合わせる
